# logcheck (linter) - Тестовое задание Selectel

`logcheck` — линтер для Go, совместимый с `golangci-lint`, который проверяет текст лог-сообщений для `log/slog` и `go.uber.org/zap`.

Он помогает поддерживать единый стиль логов и снижать риск появления в логах нежелательных данных: сообщений не на английском языке, сообщений с шумом в виде emoji и спецсимволов, а также потенциально чувствительных значений.

## Зачем это нужно

Логи быстро становятся частью публичного интерфейса системы: по ним ищут инциденты, строят алерты, пишут дашборды и разбирают ошибки в продакшене. Когда сообщения в логах оформлены по-разному, содержат лишние символы или случайно включают чувствительные данные, поддерживать систему становится заметно сложнее.

`logcheck` автоматизирует базовые проверки и делает требования к логам воспроизводимыми на уровне CI и локальной разработки.

## Что проверяет

Линтер анализирует текст лог-сообщений и проверяет, что:

- сообщение начинается со строчной буквы;
- текст сообщения написан на английском языке;
- в сообщении нет спецсимволов и emoji;
- сообщение не содержит потенциально чувствительных данных.

## Что поддерживается

### Логгеры

- `log/slog`
- `go.uber.org/zap`

### Формы вызовов

#### `log/slog`

- `slog.Info`
- `slog.Warn`
- `slog.Error`
- `slog.Debug`
- `slog.*Context`
- `slog.Log`
- `logger.LogAttrs`
- атрибуты вида `slog.String(...)`, `slog.Any(...)`, `slog.Bool(...)`, ...
- `slog.Group(...)`

#### `zap`

- `zap.Logger`
- `zap.SugaredLogger`

## Невалидные примеры

```go
slog.Info("Token validated")       // начинается с заглавной буквы
slog.Info("токен валиден")         // не английский язык
slog.Info("token validated ✅")    // emoji
slog.Info("password: 123456")      // потенциально чувствительные данные
```

## Валидный пример

```go
slog.Info("token validated")
```

## Установка
### Вариант 1. Сборка и запуск тестов локально

Используйте этот вариант, если хотите проверить проект, доработать его или собрать плагин вручную.

```bash
go test ./...
go build ./...
go build -buildmode=plugin -o ./dist/logcheck.so ./plugin
```

### Вариант 2. Использование через custom golangci-lint

Рекомендуемый вариант для реального использования в проекте — запуск через кастомную сборку golangci-lint.

```bash
golangci-lint custom -v
./bin/custom-gcl run --config .golangci.yml ./...
```

В репозитории используется конфигурация golangci-lint, совместимая с этим сценарием запуска.

### Вариант 3. Запуск через Docker

#### Сборка образа

```bash
docker build -t logcheck .
```

#### Запуск контейнера

```bash
docker run --rm -it logcheck
```

#### Запуск тестов в контейнере

```bash
docker run --rm logcheck go test ./...
```

#### Сборка плагина в контейнере

```bash
docker run --rm logcheck go build -buildmode=plugin -o ./dist/logcheck.so ./plugin
```

## Использование

После сборки custom golangci-lint линтер можно запускать так же, как и остальные проверки в пайплайне:

```bash
./bin/custom-gcl run --config .golangci.yml ./...
```

## Особенности репозитория

В текущей версии репозитория используются replace на локальные зависимости в third_party.

Это сделано для того, чтобы проект собирался воспроизводимо даже без доступа к сети. Такой подход полезен, если важно: получить стабильную сборку в изолированной среде; исключить неожиданное изменение внешних зависимостей; упростить локальную разработку и CI в средах с ограниченным доступом в интернет.
